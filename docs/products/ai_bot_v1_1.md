## 1. 产品概述

### 1.1 产品定位
一款智能对话式任务管理助手，用户通过自然语言与AI进行交互，完成个人信息管理、任务列表维护、信息搜索及深度研究等操作。产品旨在提供流畅、个性化且高效的对话式任务管理体验。

### 1.2 目标用户
- 追求高效任务管理的个人用户。
- 喜欢通过自然语言与AI交互，而非传统表单操作的用户。
- 需要AI辅助进行信息搜索、整理和研究的用户。
- 有面试准备、资料收集等场景，需要管理个人文件的用户。

### 1.3 核心价值
- **自然交互**：像与真人对话一样管理任务、获取信息。
- **个性化**：AI记住用户姓名、昵称，支持自定义AI昵称，提供亲切体验。
- **智能扩展**：集成实时搜索与深度研究，助力知识获取。
- **持续记忆**：长期记忆对话上下文，任务状态跨会话保持。
- **vibe coding式交互**：用对话代替点击，让任务管理像聊天一样自然，用户无需操作传统表单即可完成增删改查。
- **个人工作空间**：像操作系统文件夹一样管理文件，支持拖拽上传、进度反馈，方便用户整理简历、学习资料等。

## 2. 功能性需求

### 2.1 用户交互与对话界面
1. **对话式界面**：提供一个基于网页的聊天界面，支持用户与AI进行实时、连续的对话。
2. **消息展示**：清晰区分用户消息与AI助手消息，支持富文本或Markdown格式展示消息内容（如列表、链接、引用、工具调用结果等）。
3. **输入支持**：用户可以通过文本输入框发送消息，支持回车键快捷发送。
4. **富文本标记**：对于通过工具（如搜索API）或RAG获得的内容，需在AI回复中显式标记，支持鼠标悬停显示原始数据（如搜索结果元数据、检索片段）。
5. **主对话区布局（参考 DeepSeek 类产品）**：用户进入对话能力后，**页面中间以单一对话流为主**，输入区域固定在对话区底部，避免主区域被无关空白或非对话内容占据。
6. **左侧会话列表**：在侧边栏「对话」入口下提供**可折叠**的**会话（线程）列表**；用户点击其中一项时，中间主区域**加载并展示该会话下的全部历史消息**，后续发送的消息归属当前选中的会话。
7. **跨次登录与历史可见性**：用户**退出登录后再次登录**（同一身份）时，须能拉取并展示其**历史会话列表及各会话内的消息**，不得出现「登录成功但历史空白」的体验（与 §4 对话连续性假设一致）。

### 2.2 用户信息管理
1. **匿名／昵称登录**：用户通过 **显示名称（昵称）** 建立身份并登录，**无需邮箱+密码的传统注册**。该显示名称在系统内 **全库唯一**：**同一名称始终对应同一用户**；若输入的名称已存在，则视为**该用户的再次登录**（与「欢迎回来」一致），不得在同一名称下创建多个独立账号。
2. **首页字段**：登录／入口页提供 **名称（必填）** 与 **邮箱（选填）** 输入框；用户可不填邮箱直接进入产品，邮箱可在后续对话或设置中补全。
3. **主行动按钮文案**：根据名称是否已对应存量用户区分文案——**全新名称（首次创建用户）** 时按钮为 **「开始吧」**；**名称已存在（回访用户）** 时按钮为 **「欢迎回来」**（由前端根据登录接口或预检接口返回的标识展示）。
4. **第一轮对话中的信息补全**：若当前用户资料中 **姓名或邮箱仍缺失或不完整**（例如仅有机读占位、或用户未填邮箱），AI 应在 **该会话中用户的首条用户消息之后的第一条助手回复** 中，在解答用户问题的同时，**自然、友好地询问尚未掌握的姓名或邮箱**（缺什么问什么，避免重复索要已有字段）。用户补充后，系统须持久化并在后续所有对话中 **使用真实姓名称呼用户**（与下条一致）。
5. **信息存储**：系统持久化用户的显示名称、邮箱（允许为空直至补全）、AI 昵称及偏好等；再次访问时优先使用已存资料，减少重复询问。
6. **个性化称呼**：在已掌握用户姓名的前提下，AI 在后续对话中应使用该姓名称呼用户，以增强亲切感与一致性。

### 2.9 多会话标题与跨次登录
1. **会话实体**：每一条侧边栏列表项对应一个独立会话（线程），拥有自己的消息历史；用户可创建新会话或在会话间切换。
2. **自动会话标题**：当某一会话内完成 **至少一轮**「用户消息 → 助手回复」后，系统应 **自动生成** 一条简短、达意的会话标题（可由模型总结首条主题），作为列表中的默认展示名称。
3. **重命名**：用户在左侧会话列表项上 **右键**，上下文菜单中提供 **「修改名称」**；选择后支持 **原地编辑** 会话标题并保存，覆盖自动标题。
4. **与 §2.1-6、§2.1-7 的关系**：会话列表、标题与重命名能力服务于多主题并行与快速辨识；历史消息加载与再登录展示须同时满足 §2.1-7。

### 2.3 任务管理功能
用户可以采用类似 **vibe coding** 的方式，通过自然语言对话直接对任务列表及任务细节进行增删改查，无需操作传统表单。具体支持以下操作：
1. **创建任务**：添加新的任务到列表中。
2. **查询任务**：查看当前所有的任务，或按条件（如状态、关键词）筛选任务。
3. **修改任务**：更新现有任务的内容、状态（如“进行中”、“已完成”）或其他属性。
4. **删除任务**：从列表中移除已完成或不再需要的任务。
5. **细化需求**：对于列表中的单个任务，用户可以与其进行深入对话，添加、修改或删除该任务的详细描述、子任务、备注等具体需求。

### 2.4 AI个性化设置
1. **昵称设置**：允许用户为AI助手设置一个自定义的昵称。
2. **昵称修改**：允许用户在后续使用中随时修改AI的昵称。
3. **昵称使用**：AI在对话中应使用用户为其设定的昵称进行自我指代（如“你好，我是你的助手[昵称]”）。

### 2.5 个人工作空间与文件管理

#### 2.5.1 工作空间视图
用户拥有一个专属的“工作空间”，以类似操作系统文件夹浏览器的形式展示其所有上传的文件。界面包含：
- **文件列表/网格视图**：以列表或卡片形式展示文件，显示文件名、类型、大小、上传时间、语义类型标签等。
- **文件夹结构**：支持创建文件夹，用户可将文件归类整理（可选，1.0版本可简化为扁平结构，但预留扩展）。
- **操作按钮**：支持重命名、删除、下载文件。

#### 2.5.2 文件上传
- **拖拽上传**：用户可将本地文件拖拽到工作空间的任意位置（或指定文件夹），触发上传流程。
- **点击上传**：提供“上传文件”按钮，打开系统文件选择器。
- **文件类型支持**：支持图片、视频、音频、PDF、Word、Excel、文本等常见格式，但单个文件大小不得超过 **64 MB**。
- **元数据填写**：上传时弹出弹窗，要求用户填写文件的**语义类型**（如“简历”、“学习资料”、“小抄”、“项目文档”等）。语义类型用于后续RAG检索时的分类过滤。

#### 2.5.3 上传进度与状态反馈
- **初始占位节点**：用户开始上传后，在工作空间窗口中立即显示一个**虚线边框**的文件节点，表示“正在上传”。
- **进度条**：在该节点上展示实时上传进度（百分比或进度条），支持大文件上传的长时间反馈。
- **上传成功**：上传完成后，节点边框变为**实线**，并刷新文件图标、大小等信息。同时系统通过**系统通知**（浏览器通知或页面内通知）告知用户“文件上传成功”。
- **上传失败**：若上传失败，节点边框变为**红色**，并在节点下方显示一个信息框，说明失败原因（如“网络超时”、“文件过大”）。系统也通过通知告知用户失败信息。
- **失败处理**：用户可手动删除失败的节点，或点击节点上的“重试”按钮重新上传。

#### 2.5.4 文件与AI对话的联动
- 上传的文件会自动进行文本提取（若为PDF、Word等）并向量化，存入Qdrant，供RAG检索使用。
- 在对话中，AI可根据用户问题，自动检索相关文件内容，并在回复中引用（通过`<rag>`标记展示源文件）。
- 用户可通过对话指令直接管理文件，例如：“帮我删除上次上传的简历”、“把‘学习资料’文件夹里的文件都标记为‘重要’”。

### 2.6 扩展功能：搜索与深度研究

#### 2.6.1 搜索API选型调研
经调研，**Serper.dev** 是一款高性能的Google搜索API，具备以下特点，适合本项目：
- **速度快**：平均响应时间1-2秒，保证对话流畅性。
- **数据全面**：返回结构化JSON，包含网页（organic）、知识图谱（knowledgeGraph）、图片（images）、新闻（news）、视频（videos）、购物（shopping）、地图（places）、学术文章（scholar）、专利（patents）等多种类型结果。
- **成本可控**：提供免费额度，适合原型及中小规模使用，符合项目免费云资源要求。
- **易集成**：基于REST API，可通过简单的HTTP请求调用，便于AI工具调用。

#### 2.6.2 搜索功能应用设计
1. **工具注入**：将Serper.dev封装为一个可调用的工具（Tool），AI根据用户意图自动判断是否需要调用。工具支持按需指定搜索类型（如`type=news`获取最新资讯，`type=places`查找地点），默认返回有机搜索结果。
2. **搜索执行**：当用户提出需要实时或外部信息的问题时（如“今天有什么AI新闻？”、“帮我查一下量子计算的最新进展”），AI识别意图，调用Serper API，获取结构化数据。
3. **结果整合**：AI将API返回的原始数据（标题、链接、摘要、日期等）进行整理、归纳和总结，以自然、易读的方式呈现给用户，而非直接展示JSON。对于包含多种结果类型的复杂查询，AI可融合多种信息源，生成结构化回答。
4. **深度研究支持**：
   - 针对研究性课题，AI可自主规划多轮搜索。
   - **子代理规划**：主AI将研究任务拆解为若干子问题，依次调用搜索工具，收集结果。
   - **记忆召回**：每一轮搜索结果暂存于短期记忆，AI综合所有信息后，生成一份带引用来源的综述报告。
   - 支持用户后续追问，AI能基于之前的研究记忆进行针对性补充。
5. **搜索上下文管理**：
   - 对于同一主题的连续搜索，系统保留对话上下文，避免用户重复输入关键词。
   - AI能理解“上一条结果里提到的那个公司”这类指代，并基于历史搜索结果进行二次筛选或深度查询。
6. **成本与频率控制**：考虑到免费API的速率限制，系统需在后台记录用户调用次数，对高频请求进行友好提示；对于复杂研究任务，可主动询问用户是否继续。

### 2.7 上下文与记忆管理
1. **短期记忆**：AI能够在当前对话会话中记住用户刚刚提到的信息（如任务、偏好等），并据此进行连贯的对话。
2. **长期记忆**：AI能够从历史对话中提取并长期保存关键信息，例如：
   - 用户个人信息（姓名、邮箱）。
   - 用户设定（如AI的昵称）。
   - 用户创建的任务及详细需求。
   - 用户的重要偏好或习惯（如回复风格偏好）。
   - 用户上传的文件及其语义类型。
3. **记忆召回**：AI在生成回复时，应能智能地检索和利用相关的短期和长期记忆，使回复更具相关性和个性化。例如，当用户说“帮我把它标记为完成”时，AI应能结合上下文知道“它”指的是哪个任务。

### 2.8 工具调用与子代理规划
1. **工具识别**：AI能够准确理解用户的意图，并识别出完成该意图所需调用的工具或动作（如“搜索”、“添加任务”、“修改昵称”等）。
2. **自动执行**：对于识别出的工具调用，AI应能自动执行相应操作，并将操作结果反馈给用户。
3. **复杂任务规划**：对于复杂任务（如“深度研究”），AI应能自主规划执行步骤，可分解为多个子任务，并按序调用相应工具或进行多轮交互来完成。
4. **工具元信息**：在AI的可用工具列表中，明确描述每个工具的功能、输入参数、输出格式，帮助AI准确调用。
5. **异常处理**：当工具调用失败或超时，AI应告知用户并建议稍后重试，或尝试其他替代方案。

## 3. 非功能性需求

### 3.1 性能
1. **响应速度**：AI回复应具备良好的响应速度，对于简单对话和操作，应在3-5秒内给出回复。对于耗时较长的任务（如深度研究），应向用户提供进度反馈。
2. **文件上传速度**：单文件上传（≤64 MB）应在合理时间内完成，并实时更新进度条。大文件支持分片上传（可选）。
3. **并发支持**：系统应能稳定支持一定数量的用户同时访问和使用，无明显性能下降。

### 3.2 可用性与易用性
1. **界面美观**：前端界面应简洁、美观、现代，符合主流审美标准，提供良好的视觉体验。工作空间应直观，文件操作符合用户习惯。**登录／落地首页**应在不牺牲可读性的前提下，通过动效、层次、色彩与字体等体现 **AI／未来科技感**，与产品「智能助手」定位一致（具体视觉由设计与前端实现细化）。
2. **交互流畅**：用户操作应直观、自然，对话交互应尽可能接近真人交流，减少用户的学习成本。
3. **引导清晰**：对于初次使用的用户，AI应提供清晰的引导，帮助用户快速了解产品的核心功能（如任务管理、AI昵称设置、文件上传等）。
4. **容错性**：当用户输入模糊或无法处理时，AI应能友好地提示用户，并引导用户提供更清晰的信息，而非直接报错或中断对话。文件上传失败时，应提供明确的重试机制和错误原因。

### 3.3 安全性与隐私
1. **数据隔离**：不同用户之间的数据（个人信息、任务列表、上传文件等）应严格隔离，确保用户只能访问和操作自己的数据。
2. **数据持久化安全**：用户存储的个人信息（如邮箱）和任务数据应得到妥善保存，防止数据丢失或未授权访问。
3. **搜索数据保护**：用户搜索的关键词及返回结果不应对其他用户可见，不进行跨用户的搜索分析或用于其他目的。
4. **外部依赖隐私**：调用Serper.dev时，仅传输必要的查询参数，不附带用户身份信息（如姓名、邮箱）。
5. **文件存储安全**：上传文件存储于私有R2桶，通过签名URL访问，确保只有所有者可访问。

### 3.4 可扩展性
1. **模块化**：产品的功能模块（如对话管理、用户管理、任务管理、文件管理、搜索服务等）应在逻辑上保持清晰和独立，以便于未来功能的增加或修改。
2. **功能可插拔**：未来新增工具或功能（如接入新的搜索API、日历集成等）应能方便地集成到现有体系中。
3. **搜索服务封装**：将Serper.dev的调用逻辑独立为模块，便于未来替换为其他搜索API或增加本地搜索缓存。

### 3.5 可靠性
1. **服务可用性**：核心对话服务应保持高可用性，尽量减少因外部依赖（如AI模型、搜索API）问题导致的服务中断。
2. **数据一致性**：用户对任务列表的增删改查操作应保证数据的一致性，避免出现数据冲突或状态不同步的情况。
3. **搜索服务降级**：若Serper.dev服务不可用，系统应能优雅降级，由AI告知用户“搜索服务暂时不可用，请稍后再试”，不中断对话其他功能。
4. **文件上传鲁棒性**：支持网络中断恢复、自动重试，避免因临时故障导致上传失败。

## 4. 约束与假设
- **用户数据存储**：假设用户信息（显示名称、可选邮箱）和任务数据可以被可靠地持久化存储，并能与**多会话对话历史**、长期记忆进行关联。
- **名称唯一性**：显示名称在库内全局唯一，作为匿名登录主键；与邮箱是否必填解耦。
- **对话连续性**：假设系统能够维护 **多个会话线程** 及各自消息历史，并确保用户在**刷新页面、切换会话或退出后再次登录**后，能恢复会话列表与各会话内容（至少个人信息、任务列表与历史消息不丢失）。
- **外部服务依赖**：核心功能（智能对话、信息搜索）依赖于外部AI服务和搜索服务的稳定性和可用性。
- **免费资源限制**：使用免费层级的AI模型和搜索API，需遵守相应速率限制和配额。
- **文件大小限制**：单文件上传上限64MB，超过此大小的文件将被拒绝，并提示用户压缩或使用外部存储。

## 5. 验收标准

### 5.1 用户引导与识别
- **场景**：资料不全时，用户在**该会话第一条助手回复**中即被自然询问缺失的姓名或邮箱；资料齐全后，AI 能正确称呼用户。
- **验收**：用仅有显示名、无邮箱的账号发首条消息，验证助手在同一条回复中既回应诉求又询问邮箱（或缺失项）；补全后再次对话，验证称呼一致。

### 5.9 多会话匿名登录与首页
- **场景 A（布局）**：登录后中间为单列对话区；左侧「对话」下可折叠会话列表，点击某项加载该会话历史。
- **验收 A**：折叠／展开列表、切换会话后消息区内容与发送归属正确。
- **场景 B（再登录）**：用户退出后使用同一名称再次登录，历史会话与消息均可见。
- **验收 B**：手动执行登出／登录，核对列表与消息与登出前一致。
- **场景 C（标题）**：新会话完成首轮问答后列表中出现自动标题；右键「修改名称」可原位改名并持久化。
- **验收 C**：自动标题合理可辨；改名后刷新仍存在。
- **场景 D（登录页）**：新名称显示「开始吧」，已存在名称显示「欢迎回来」；**同名再次提交**应进入该既有账号而非报错；邮箱可空提交。
- **验收 D**：分别用新名称首次进入、同名称再次进入验证按钮文案与账号一致性；邮箱留空可登录。
- **场景 E（首页视觉）**：落地页具备明确的科技感／未来感表达（动效或视觉层次），且不干扰表单操作。
- **验收 E**：设计／产品走查通过即可。

### 5.2 任务管理闭环
- **场景**：用户能通过对话成功创建、查看、修改和删除任务及其详情。
- **验收**：执行一系列任务操作，确认任务列表状态与对话反馈一致。

### 5.3 AI个性化
- **场景**：用户可以成功为AI设置昵称，AI后续对话中使用该昵称。
- **验收**：设置昵称后，AI在后续回答中以该昵称自我指代。

### 5.4 搜索与研究
- **场景**：用户提出“今天有什么科技新闻”时，AI能返回基于Serper.dev新闻搜索的整合结果，并包含来源链接。
- **验收**：手动测试，验证搜索结果包含新闻标题、来源、日期，并支持点击链接。

### 5.5 深度研究
- **场景**：用户提出“帮我研究一下太空旅游的现状”时，AI能自主进行至少2次不同角度的搜索，并生成包含技术进展、主要公司、市场前景等部分的简要报告。
- **验收**：验证AI回复中至少包含两次工具调用标记，最终报告结构清晰、信息整合合理。

### 5.6 工具调用
- **场景**：在对话中，AI能准确识别需要实时信息的意图，并自动调用搜索工具，整个过程对用户透明（或通过“正在搜索…”等状态提示）。
- **验收**：测试多个意图，确认AI正确调用对应工具，并展示标记及元数据。

### 5.7 记忆能力
- **场景**：在对话中，AI能正确引用之前提到的任务或信息。
- **验收**：用户说“把刚才那个任务改成已完成”，AI能准确识别并执行。

### 5.8 个人工作空间与文件管理
- **场景**：用户上传一份PDF简历，填写语义类型“简历”，上传过程中显示进度条，完成后文件节点变为实线，并在对话中可被AI检索引用。
- **验收**：
  - 拖拽或点击上传功能正常，上传时显示虚线节点和进度条。
  - 上传成功/失败有相应的UI变化和系统通知。
  - 上传失败后可重试或删除节点。
  - 上传的文件可在工作空间列表中查看，并支持重命名、删除、下载。
  - 在对话中提问“我的简历内容是什么？”AI能通过RAG检索到该文件内容并回答。

### 7.1 术语表
| 术语 | 说明 |
|------|------|
| **RAG** | 检索增强生成，通过检索相关文档片段来增强AI生成内容的准确性和上下文。 |
| **工具调用** | AI模型识别用户意图后，调用外部API或函数执行特定操作的过程。 |
| **子代理规划** | AI将复杂任务分解为多个子任务，并依次调用相应工具或子模型完成。 |
| **Serper.dev** | 一款提供快速、廉价Google搜索结果的API服务，支持多种结果类型。 |
| **长期记忆** | 跨会话持久化存储的用户信息、任务、偏好等数据。 |
| **短期记忆** | 当前对话会话中临时存储的上下文信息。 |
| **会话（线程）** | 侧边栏中的一项，对应独立消息历史；用户可在多个会话间切换。 |
| **显示名称** | 用户登录时输入的、全库唯一的昵称／代号，用作匿名身份主键。 |
| **工作空间** | 用户管理个人文件的界面，类似操作系统文件夹浏览器。 |
| **语义类型** | 用户为上传文件标记的分类标签，如“简历”、“学习资料”等，用于增强检索。 |
